1. Project Overview

SmartPlot Portal is a full-stack real estate web application prototype built for property listing, plot discovery, and buyer reservation flow. It allows users to sign up, log in securely, browse live plot inventory, search and filter plots, and reserve available plots under their own account.

The current version combines both buyer-facing booking features and internal inventory management features in one portal to demonstrate the complete workflow.

2. Objective

The goal of the project is to simulate a real estate platform where:

  • Users can create accounts and log in securely.
  • Plots can be listed and managed in a structured system.
  • Buyers can browse available plots and reserve them.
  • The booking lifecycle can be tracked from available to reserved.

User Management

Plot Management

  • Edit plot listings
  • Delete plot listings
  • Search plots
  • Filter by status
  • Add image URLs to plots

Booking Flow

  • Users can book an available plot
  • Booked plots move from AVAILABLE to RESERVED
  • Booked plots are tied to the logged-in user
  • Users can view My booked plots
  • Users can cancel their own booking

5. Core Workflow

  1. User opens the portal.
  2. User signs up or logs in.
  3. Backend validates credentials and returns a JWT token.
  4. Frontend stores the token and uses it in future requests.
  5. User browses available plots.
  6. User books a plot.
  7. The plot status changes to reserved and appears under that user’s bookings.

6. Main APIs

Public Endpoints
  • POST /signup
  • POST /login
  • GET /health
Protected Endpoints
  • GET /users
  • GET /plots
  • POST /plots
  • PUT /plots/{plotId}
  • DELETE /plots/{plotId}
  • POST /plots/{plotId}/book
  • POST /plots/{plotId}/cancel-booking
  • GET /plots/my-bookings

9. Questions You May Be Asked

What problem does this solve?

It combines property listing management and buyer plot reservation in one portal.

Why Spring Boot?

It provides fast backend development, REST APIs, security support, and easy database integration with JPA.

Why MySQL?

Because this application deals with structured relational data like users, plots, and bookings.

How does authentication work?

User logs in, backend verifies credentials, returns a JWT token, and protected requests include that token.

How does booking work?

A user can book a plot only if its status is available. Booking changes the plot to reserved and ties it to the user account.

What happens if a plot is already reserved?

The backend prevents booking again unless the plot is available.

Is this production ready?

No. It is a working MVP/prototype, not a final production deployment.
